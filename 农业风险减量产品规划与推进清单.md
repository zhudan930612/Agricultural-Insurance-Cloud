# 农业风险减量产品规划与推进清单

## 1. 文档说明

### 1.1 文档定位
本文档承接《农业风险减量平台 产品上下文》中不宜展开的执行层内容，用于沉淀收费细化、技术验证、待确认项分级、责任分工和推进节奏。

### 1.2 与上下文文档的分工
- 产品上下文：回答“这是什么产品、为什么做、边界在哪里”
- 本文档：回答“下一步怎么推进、哪些事项未确认、谁来确认、按什么节奏推进”

### 1.3 当前版本
- 版本：V1.1
- 更新日期：2026-03-31

### 1.4 本轮修订重点
- 强化“首单作战定义”，避免方案在关键前提未坐实时呈现为可直接执行
- 将一期范围改为“有前提条件的建议范围”，不再默认视为已冻结
- 将收费模型从“分层设计”调整为“对外报价假设 + 内部收入判断”分离表述
- 对“天空地一体”补充资源可调用性分层，避免把理论资源写成交付能力

## 2. 首单作战定义

### 2.1 当前最关键的五个问题
- 谁买单：太保新疆项目方、太保农险条线、还是其他保险公司试点客户
- 买什么：平台、招投标支撑包、精准核保服务包、试点交付项目，还是其组合
- 凭什么买：资源整合能力、招投标支撑能力、核保交付能力、还是成本优势
- 怎么验收：是方案支撑赢标、试点交付完成、任务闭环跑通，还是形成实际核保成果
- 交付资源从哪来：卫星、无人机、地面信息员各自的真实调用方式、响应时效和授权边界

### 2.2 当前阶段判断口径
- 太保新疆：当前只能确认是重点推进机会，尚不能视为需求、预算、验收口径均已锁定的成熟项目。
- 国家减灾中心合作：当前只能表述为合作基础或合作推进中，不能默认等同于正式授权和稳定资源供给。
- 天空地资源：当前应拆分为“今日可调”“申请后可调”“理论可用”三类，不应统一表述为可交付能力。
- 一期范围：在首单买单对象、交付物和验收口径未确认前，只能是建议范围，不能视为冻结范围。

### 2.3 首单作战定义建议

| 项目 | 建议写法 | 当前状态 |
|---|---|---|
| 首单目标 | 先拿下可验证交付的小闭环，不以“大而全平台成交”为默认前提 | 待确认 |
| 优先售卖对象 | 优先假设为“招投标支撑包”或“精准核保服务包”，平台能力作为支撑资产 | 待确认 |
| 最小成交单元 | 可先成交单项服务或试点项目，不强绑定整平台建设 | 建议采用 |
| 最小验收单元 | 至少需要明确“任务可执行 + 结果可回传 + 证据可复核”三项中的主验收项 | 待确认 |
| 冻结一期前提 | 买单方、购买内容、资源供给、验收标准四项至少确认 80% | 待确认 |

### 2.4 首单验收口径建议
- 弱验收：数据导入成功、地图可展示、任务可下发、结果可回传
- 中验收：完成一个区域/批次的核保或核查闭环，并形成可复核证据链
- 强验收：直接支撑招投标、形成试点案例或帮助客户完成业务目标

说明：
- 当前不建议把“帮助太保赢标”直接作为唯一验收口径，因为其受外部变量影响过大。
- 更稳妥的做法是先锁定“中验收”作为一期核心目标，再将“强验收”作为商务目标。

## 3. 收费模式细化

### 3.1 当前收费结构判断
当前收费模型仍属于业务推演，不应视为已验证报价策略。现阶段更适合将其拆成“对外报价假设”和“内部收入判断”两层。
- 对外报价假设：客户最终可能购买的是招投标支撑包、精准核保服务包、试点交付项目、平台接入项目中的一种或组合
- 内部收入判断：首阶段更可能由核保/核查类服务费形成收入主体，平台建设费和报告类收入为补充

当前不确定点：
- 客户是否真的为“平台”付费，尚无坐实证据
- 风险减量服务是否形成独立续费收入，当前缺乏成交证据支持
- 首单最可能成交的形态，可能不是“平台+服务打包”，而是单项服务包或试点包

### 3.2 对外报价结构假设

| 报价形态 | 适用场景 | 当前判断 | 风险提示 |
|---|---|---|---|
| 招投标支撑包 | 客户仍处于方案比选和投标阶段 | 首单高概率形态之一 | 容易被误判为低客单价售前支持 |
| 精准核保服务包 | 客户先购买可执行服务能力 | 首单高概率形态之一 | 依赖资源调度和交付组织能力 |
| 试点项目交付包 | 客户希望先在区域内试运行 | 较符合当前项目成熟度 | 需明确试点范围和验收口径 |
| 平台建设与接入项目 | 客户已明确长期建设意图 | 当前不宜默认为首单主形态 | 前提条件最多、成交周期最长 |
| 风险减量年度服务包 | 客户已认可平台持续价值 | 更适合作为二阶段报价项 | 当前续费逻辑尚未被验证 |

### 3.3 内部收入结构判断
- 核保/核查服务费：当前最可能形成首阶段主要收入来源
- 招投标支撑与专项报告费：可作为首单前置切入收入
- 平台建设费：更可能出现在客户确认长期合作之后
- 风险减量续费收入：目前只是潜在方向，尚缺少客户付费意愿证据

### 3.4 定价口径修正
- 现阶段不建议保留“20-50 万/省级项目起”这类具体区间作为正式判断，除非已有明确来源依据。
- 更稳妥的写法应为“平台建设与接入费需结合部署范围、交付内容和客户预算单独建模”。
- 卫星、无人机、人工核查的单价逻辑可以保留，但应统一标注为“待商务建模”。

### 3.5 成交案例状态
- 太保新疆项目：当前为重点推进机会，尚不能表述为已成交案例。
- 其他保险公司项目案例：待补充，如未形成正式合同或验收闭环，应统一标注为"待商务确认/待案例沉淀"。

## 4. 技术成熟度与验证

### 4.1 技术成熟度评估

| 技术能力 | 当前成熟度 | 当前判断依据 | 主要风险/限制 | 是否需要专项验证 |
|---|---|---|---|---|
| 卫星遥感识别耕地/作物大类 | P1 可用 | 行业内已有较多遥感识别应用，适合作为大范围初筛和底图识别 | 受云层、时相、作物生长阶段影响，精度受区域差异影响明显 | 需要 |
| 卫星遥感直接识别具体作物类型 | P2 谨慎评估 | 理论可行，但对样本、区域模型和时序影像依赖高 | 跨省泛化能力弱，误识别会直接影响核保可信度 | 强烈需要 |
| 基于遥感估算种植面积 | P1 可用 | 适合做面积估算、异常筛查和趋势对比 | 难替代争议地块的最终人工核验 | 需要 |
| 病虫害智能识别 | P2 谨慎评估 | 适合作为预警线索，不宜直接作为定责定损依据 | 病虫害类型复杂，需结合地面巡查验证 | 强烈需要 |
| 无人机精细核查 | P1 可用 | 技术成熟，适合作为重点区域核查手段 | 调度成本、空域审批、天气影响较大 | 需要 |
| 地面信息员现场核验 | P1 可用 | 最接近真实业务落地方式，可补足遥感盲区 | 人员组织、培训、真实性管控是关键 | 需要 |

### 4.2 对外统一技术口径
- 可卖能力：遥感初筛、无人机复核、人工核验、理赔核查取证、地图化展示、基础风险分析
- 规划能力：作物类型智能识别、病虫害智能识别、自动定损、复杂模型化风险区划
- 禁止误写口径：当前不得在对外材料中将病虫害能力写成“精准识别”“自动定损依据”或“高精度稳定识别”

### 4.3 技术验证结论
- 现阶段更适合将"卫星遥感识别作物类型""病虫害智能识别"表述为辅助判断能力或规划能力，不宜直接承诺为高精度、全区域稳定可用能力。
- 一期产品应优先承诺"遥感初筛 + 无人机复核 + 人工核验"的组合交付模式。
- 后续需补充试点验证指标，包括识别准确率、召回率、误报率、覆盖区域、验证样本量和人工复核差异率。

### 4.4 资源可调用性分层

| 资源类型 | 当前应采用的判断口径 | 需要补齐的证据 |
|---|---|---|
| 卫星遥感 | 优先判断为“申请后可调/按授权获取” | 数据授权、获取周期、可用区域、费用 |
| 无人机 | 优先判断为“申请后可调” | 调度 SLA、空域审批、回传周期、单次成本 |
| 灾害信息员 | 优先判断为“理论可用或局部可调” | 可调度规模、管理权限、真实性控制机制 |

说明：
- 在未拿到明确机制前，不建议统一写成“今天就能调”。
- “天空地一体”可以保留为目标交付模式，但必须区分目标能力和当下已掌握能力。

### 4.5 建议验证任务
- 选取 1-2 个目标试点区域，完成作物识别和面积估算小样本验证
- 形成卫星识别结果与人工核验结果对比样本
- 验证病虫害识别能否达到“预警线索”级别可用
- 明确无人机调度时效、飞行组织要求和回传周期
- 明确地面信息员的取证真实性控制措施，如水印、定位、防篡改和抽检复核机制

## 5. 待确认项分级与推进

### 5.1 待确认项分级表

| 优先级 | 待确认项 | 责任建议 | 建议截止时间 | 影响范围 |
|---|---|---|---|---|
| P0 | 太保新疆项目所处阶段（口头机会/联合推进/意向确认/需求已锁定） | 商务负责人 | 一期范围讨论前确认 | 决定首单判断是否成立 |
| P0 | 太保合作协议的签署进度和具体合作范围 | 商务负责人 | 招投标方案输出前确认 | 直接影响首单落地和方案边界 |

| P0 | 太保新疆项目的必选功能范围与验收口径 | 产品负责人 + 商务负责人 | 需求范围冻结前 | 直接影响一期范围 |
| P1 | 灾害信息员能否纳入协保任务，以及管理权限归属 | 产品负责人 + 商务负责人 | 一期方案评审前 | 影响地面执行组织模式 |
| P1 | 与保险公司核心系统对接的技术方案 | 技术负责人 | 技术设计前 | 影响导入效率和系统边界 |
| P1 | 遥感识别作物类型、面积测算、病虫害识别的技术验证结果 | 技术负责人 + 产品负责人 | 技术验证结束后 | 影响智能化承诺边界 |
| P1 | 新疆以外的省份试点计划和时间节点 | 商务负责人 | 首单立项后 1 个月内 | 影响复制路径和资源准备 |
| P2 | 产品正式名称是否采用"农保云" | 产品负责人 | 对外品牌材料输出前 | 影响品牌一致性 |
| P2 | 是否需要开发农户端/投保人端 | 产品负责人 | 二期规划前 | 影响产品边界和研发投入 |
| P2 | 勾地块功能的业务优先级和落地范围 | 产品负责人 | 一期复盘后 | 影响中长期数据资产路线 |

### 5.2 分级说明
- P0：不确认就无法稳妥对外承诺或无法冻结一期方案
- P1：影响交付质量和中期方案可信度
- P2：可在一期之后继续判断

### 5.3 周度推进建议
1. 商务与法务优先处理 P0 中的合作边界和授权问题。
2. 产品与技术并行处理 P0/P1 中的首单定义、一期范围和技术验证问题。
3. 每周更新一次责任人、状态、风险和下一动作。

## 6. 一期推进建议

### 6.1 一期范围前提
只有在以下前提满足后，才建议冻结一期范围：
- 买单对象已明确
- 首单购买内容已明确
- 首单验收口径已明确
- 资源供给方式已明确

### 6.2 一期建议范围
- 分级核保流程闭环
- 理赔核查任务与结果回传
- 面向招投标或试点交付的结果展示能力
- 保单地图化基础底座（前提：它直接服务核保执行、结果展示或招投标表达）

说明：
- “保单地图化”不应因为架构上合理就自动成为一期必做，只有在它直接支撑首单成交或首单交付时，才应保留在一期。
- “基础风险区划展示”如主要用于售前展示，应降级为辅助能力，不与核保闭环同优先级。
- 一期若不做全面勾地块，也应评估是否保留“争议地块坐标/边界记录”这一最低限度能力，以支撑精准核保说服力。

### 6.3 非一期必做
- 农户端
- 勾地块精细功能全面铺开
- 病虫害高精度智能识别承诺
- 全国化复制方案细化
- 复杂平台化经营分析能力
- 未被首单明确购买的长期风险减量续费能力
